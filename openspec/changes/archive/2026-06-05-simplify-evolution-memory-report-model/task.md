## Task Goal

Define and implement the new close-time contract for project evolution, study memory, and derived research-map rendering without reintroducing separate report or boundary-governance layers.

## Study Link

This task supports the study question about simplifying QDD's project-state model around `evolution.yaml`, `context/memory/`, and `research-map.html`.

## Method

Use the existing managed-file contract layer as the source of truth, then align runtime defaults, close-time writes, status/instructions outputs, visualization rendering, and validation around the simplified model.

The implementation should proceed in this order:

1. tighten the file contracts for evolution and memory
2. update close-time runtime writes
3. update status/instructions/inspection surfaces
4. update research-map rendering
5. update prompts and docs that still describe the old semantics

## Expected Outputs

- updated file contracts for:
  - `evolution.yaml`
  - `context/memory/STUDY-XXX.md`
- updated runtime code for close-time state writes
- updated `qdd status --json` and `qdd instructions --json` behavior
- updated `research-map.html` derivation logic
- updated tests covering the simplified model
- updated documentation/examples generated by the managed contract layer

## Run Contract

Each implementation run must make provenance explicit for:

- which contract files changed
- which runtime surfaces changed
- which prompt or documentation files changed
- which tests were added or updated
- whether older fields or report assumptions were removed cleanly

## Failure / Blocker Conditions

Progress is blocked or invalid if:

- runtime behavior still depends on `question_before` or `question_after`
- memory and report both remain default close-time outputs
- `research-map.html` still requires state that does not exist in the new `evolution.yaml`
- the new model only works by layering compatibility patches over the older semantics
